    # 1.说一下Http与Https

    1. http是指超文本传输协议，是互联网上应用最为广泛的网络协议，是一个客户端与服务端请求与应答的标准（TCP）,用于从WWW服务器传输超文本到本地浏览器的传输协议，目的让浏览器更加高效，减少网络传输
    2. https是指http的安全版，即在http下加入了ssl层，其安全基础也是ssl。该协议的主要作用就是建立一个信息安全通道，确保数据传输，保证网站真实性。
    3. 两者区别就是数据的加密，http传输的数据是未加密的，而https通过设置ssl层来对传输的数据进行加密处理，简单来说https协议是由http和ssl协议构建的可进行加密传输和身份认证的网络协议。
    主要区别有：
1.https协议需要ca证书，费用高
2.两者使用不同的链接方式，端口也不同，一般http协议的端口是80，https而为443
3.http的连接简单，无状态;https协议是由ssl+http协议构建的可进行加密传输、身份认证的网络协议

    • https协议的工作原理：

      1.客户端通过https url访问服务器，要求与web服务器建立ssl链接。

    # 2.Tcp三次握手，一句话概括

    客户端和服务端都需要直到各自可收发，需要进行三次握手

    一句话：客户端发送请求连接，服务端确认连接且发起连接，客户端再确认

    每次握手的作用

    • 第一次握手：S可以确认自己可接受C发送的报文段
    • 第二次握手：C可以确认S收到了自己发送的报文段，并且可以确认自己可接受S发送的报文段
    • 第三次握手：S可以确认C收到了自己发送的报文段

    # 3.TCP和UDP的区别

    TCP指传输控制协议，是一种面向连接的、可靠的、基于字节流的传输层通信协议

    UDP是指用户数据报协议，给应用程序提供一种无需建立连接就可以发送封装ip数据包的方法

    • TCP面向连接，而UDP无连接即发送数据
    • TCP连接传送的数据无差错，不丢失，不重复，且按序到达；UDP不保证可靠交付
    • TCP面向字节流，UDP面向报文，所以网络拥塞不会使得发送速率降低（因此出现丢包）
    • TCP只能是1对1，UDP在这基础上还支持1对多
    • TCP首部较大为20字节，而UDP只有8字节
    • TCP可靠；UDP不可靠

    # 4.WebSocket的实现和应用

    什么是WebSocket

    WebSocket是一种在单个TCP连接上进行双全工通信的协议，是HTML5的协议，它使得客户端与服务端之间的数据交换变得更加简单，创建持久性连接。

    是什么样的协议，有什么优点

    HTTP的生命周期通过Request来界定，即一个Request一个Response，那么在Http1.0协议中这次Http请求就结束了。在Http1.1中改进了协议，多了一个Connection：Keep-alive，也就是说在一个Http连接中可以发送多个Request接收多个Response，其中Response是被动的不能主动发起。

    WebSocket基于Http协议，也可以说借用了Http协议来完成一部分的握手，在握手阶段与Http是相同的，而WebSocket握手协议的实现基本是通过2个属性：upgrade，connection。

    # 5.HTTP请求的方法

    1. GET：用于请求指定页面信息，并返回实体主体
    2. HEAD：类似于get请求，只不过返回的响应中没有具体的内容，用于获取报头
    3. POST：向指定资源提交数据进行处理请求。
    4. PUT：从客户端向服务器传送的数据取代指定的文档的内容
    5. DELETE：请求服务器删除指定页面
    6. CONNECT：HTTP1.1协议中预留给能够将连接改为管道方式的代理服务器
    7. OPTIONS：允许客户端查看服务器的性能
    8. TRACE：回显服务器收到的请求，用于测试和诊断

    # 10.说一下Http2.0

    http2.0是基于1999年发布的http1.0之后的首次更新，相比于1.0提升了访问速度，允许多路复用。

    多路复用是指：允许同时通过单一的HTTP/2连接发送多重请求-响应信息。

    改善了在http1.1中，浏览器客户端在同一事件，针对同一域名下的请求有一定数量的限制。

    并且HTTP2.0会将所有的传输信息分割成更小的信息或者帧，并进行二进制编码。

    # 12.fetch发送两次请求的原因

    Fetch API (opens new window) 提供了一个 JavaScript 接口，用于访问和操纵 HTTP 管道的一些具体部分，例如请求和响应。

    那么fetch()发送两次请求且第一次发送状态码为204的原因是：在用fetch第一次发送一个options请求询问服务器是否支持修改的请求头，若支持则在第二次中发送真正请求。

    # 13.Cookie、sessionStorage、localStorage

    三者都是保存在浏览器端的，并且都是同源

    cookie：可以在浏览器和服务器端来回传递，存储容量 小，只有大约 4K 左右，而另外两个不会自动把数据发给服务器。

    sessionStorage：本身就是一个回话过程，关闭浏览器后消失，session 为一个回话，当页面不同即使是同一页面打开两次，也被视为同一次回话

    localStorage：同源窗口都会共享，并且不会失效，不管窗口或者浏览器关闭与否都会始终生效

    cookie的作用很常见例如保存用户的登录状态，设置过期时间。
跟踪用户行为，例如记住上次天气预报查询的地区
便于定制页面，cookie记录用户的风格页面选项

    # 14.Web worker

    在 HTML 页面中，如果在执行脚本时，页面的状态是不可响应的，直到脚本执行完成后， 页面才变成可响应。web worker是运行在后台的 js，独立于其他脚本，不会影响页面你的性能。并且通过postMessage将结果回传到主线程。这样在进行复杂操作的时候，就不会阻塞主线程了。

    # 16.iframe是什么，有什么缺点

    iframe标签：规定一个内联框架，创建包含另一个文档的内联框架

    缺点：

    • 会阻塞主页面的 onload 事件
    • 搜索引擎无法解读这种页面，不利于SEO（搜索引擎优化）
    • iframe和主页面共享连接池，而浏览器对相同区域有限制所以会影响性能。

    # 17.Doctype作用？严格模式与混杂模式？意义

    Doctype声明位于文档最前端，来告诉浏览器以何种方式进行页面的渲染。

    它有两种模式，分别是严格模式和混杂模式。

    **严格模式：**以该浏览器支持的最高标准运行

    **混杂模式：**向后兼容，模拟老版浏览器，防止浏览器无法兼容页面

    # 19.Cookie与session的区别

    cookie数据保存在客户端，session数据保存在服务端

    HTTP 是一个无状态协议，因此 Cookie 的最大的作用就是存储 sessionId 用来唯一标识用户

    session： 客户端每次请求服务器的时候会发送当前会话sessionid，服务器通过sessionid判断相应用户数据标志，来确定用户是否登录或者具有某种权限。数据是在服务器上面所以不能伪造

    cookie： 数据保存在浏览器端，比如登录之后服务器设置了cookie用户名，那么当再次请求服务器的时候，浏览器就会将用户名这一块发送给服务器，并且服务器会解释为cookie变量。

    1. cookie 数据存放在客户的浏览器上，session 数据放在服务器上。
2. cookie 不是很安全，别人可以分析存放在本地的 COOKIE 并进行 COOKIE 欺骗
考虑到安全应当使用 session。
3. session 会在一定时间内保存在服务器上。当访问增多，会比较占用你服务器的性能
考虑到减轻服务器性能方面，应当使用 COOKIE。
4. 单个 cookie 保存的数据不能超过 4K，很多浏览器都限制一个站点最多保存 20 个

    # 22.click在ios上有延迟，原因及如何解决

    原因

    ios搭载的safari，为了能把PC端大屏幕的页面以较好的效果展示在小屏幕手机端上，采用了双击缩放（double tap to zoom）的方案。浏览器怎么区分用户是只要单击操作还是要双击进行缩放操作。所以设置了300ms的延迟时间，加以判断用户是单击还是双击缩放。

    解决方案：

    • 粗暴型，禁止缩放：

      <meta name="viewport" content="width=device-width, user-scalable=no">
      1

    • 利用FastClick： 检测到 touchend 事件后，立刻出发模拟 click 事件，并且把浏览器 300 毫秒之后真正出 发的事件给阻断掉

    # 24.http返回的状态码

    类别 原因短语
    1XX 信息性状态码 接受的请求正在处理
    2XX 成功状态码 请求正常处理完毕
    3XX 重定向状态码 需要进行附加操作来完成请求
    4XX 客户端错误状态码 服务器无法处理请求
    5XX 服务器错误状态码 服务器处理请求出错

    2XX常见

    • 200 OK：请求已经正常处理
    • 204 No Content：请求处理成功，但没有任何资源可以返回给客户端
    • 206 Partial Content：是对资源某一部分的请求

    3XX常见

    • 301 Moved Permanently：永久性重定向
    • 302 Found：临时性重定向
    • 303 See Other：资源的URI已更新，你是否能临时按新的URI访问
    • 304 Not Modified：资源已找到，但未符合条件请求
    • 307 Temporary Redirect：临时重定向

    4XX常见

    • 400 Bad Request：服务器端无法理解客户端发送的请求
    • 401 Unauthorized：该状态码表示发送的请求需要有通过HTTP认证
    • 403 Forbidden：不允许访问那个资源
    • 404 Not Found：服务器上没有请求的资源

    5XX常见

    • 500 Internal Server Error：貌似内部资源出故障了
    • 503 Service Unavailable：抱歉，我现在正在忙着

    # 25.强缓存与协商缓存

    强缓存： 状态码为200 不会发送请求到服务器，直接从缓存中存取（http请求头可能出现expires/cache-control）

    协商缓存： 状态码为304 会通过服务器来告知缓存是否可用（http请求头可能出现Last-Modifyied//If-Modified-Since，Etag/If-None-Match）

    # 26.谈论一下304

    304

    如果客户端发送了一个带条件的 GET 请求且该请求已被允许，而文档的内容（自 上次访问以来或者根据请求的条件）并没有改变，则服务器应当返回这个 304 状态码

    # 27.前端优化

    • 降低请求量：合并资源，减少 HTTP 请求数，minify / gzip 压缩，webP，lazyLoad。
    • 加快请求速度：预解析，减少域名数，并行加载，CDN 分发。
    • 缓存：HTTP 协议缓存请求，离线缓存 manifest，离线数据缓存 localStorage。
    • 渲染：JS/CSS 优化，加载顺序，服务端渲染，pipeline。

    # 28.GET和POST的区别

    • get是从指定的资源请求数据，post是向只当的资源提交要被处理的数据
    • get参数通过url，post放在request body中
    • get在url中传递的参数有长度限制，而post没有
    • get比post危险，因为参数直接暴露在url中
    • get请求只能进行url编码，而post可进行多种
    • get请求浏览器会主动缓存
    • get请求的参数会被保存在浏览记录里，而post的参数不会
    • get产生一个tcp数据包，post产生两个tcp数据包
    对于 GET 方式的请求，浏览器会把 http header 和 data 一并发送出去，服务器响应 200
（返回数据）；
而对于 POST，浏览器先发送 header，服务器响应 100 continue，浏览器再发送 data，服
务器响应 200 ok（返回数据）。

    # 29.301与302的区别

    字面上的区别就是 301 是永久重定向，而 302 是临时重定向。 301 比较常用的场景是使用域名跳转。302 用来做临时跳转 比如未登陆的用户访问用户 中心重定向到登录页面

    1. 输入url首先寻找url中的服务器ip
    2. 浏览器会先寻找缓存，查询是否有记录：浏览器缓存—系统缓存—路由缓存
    3. 缓存中找不到就查找系统hosts文件中是否有记录
    4. 若hosts文件中没有则查询DNS服务器，得到服务器的ip
    5. 得到ip以及相应的端口号，构造http请求
    6. 请求报文中包括这次请求的信息例如请求方法、请求说明等
    7. 并将这个http请求封装到tcp包中，这个包会一次经过传输层、网络层、数据链路层、物理层直到服务器
    8. 服务器会解析这些请求作出相应返回相应的html
    9. 浏览器根据html的树形结构构建DOM树，再根据CSS样式来构建CSS对象模型树CSSOM树
    10. 构建完成后和DOM树合并为渲染树，之后进行布局 确认尺寸和位置
    11. 然后渲染页面，在解析DOM的过程中会对资源进行下载
    12. 此外还会关注缓存，先请求设置了缓存的数据，会先查看是否过期，若没过期则使用本地缓存
    13. 若过期了则请求并在服务器校验文件是否需要修改

    # 33.HTTP首部

    http的首部主要有通用首部、请求首部、响应首部、实体首部

    通用首部：表示通用信息，例如date表示报文创建时间

    请求首部：请求报文独有，如cookie，和缓存相关的if-Modified-Since

    响应首部：响应报文独有，如set-cookie，重定向相关的location

    实体首部：描述实体部分，如allow描述可执行的请求方法，content-type描述主题类型，content-Encoding描述主题的编码方式

    # 34.HTTP2.0特性

    • 内容安全，http2.0是基于https的，并且通过http2.0可以避免单纯使用https的性能下降
    • 二进制格式，http2.0采用二进制格式编码，基于二进制可以让协议有更多的扩展性，而http1.x是基于文本的
    • 多路复用，每个request请求可以随机的混杂在一起，接收方可以根据request的id将request再归属到各自不同的服务端请求里面，并且多路复用支持流的优先级，运行客户端告诉服务器哪些内容是更优先级的资源

    # 35.浏览器生成页面，会生成的树

    DOM树（遍历文档节点）和CSSOM规则树（浏览器解析CSS文件生成）

    # 36.csrf和xss的网络攻击及防范

    CSRF

    跨站请求伪造，攻击者盗用用户的身份，以用户名义发送恶意请求。CSRF能够做的事情包括：以你的名义发送邮件，发信息，盗取你的账户，甚至于购买商品，虚拟货币的转账等等...

    防御方式：使用验证码，检查https头部的refer，使用token

    XSS

    跨站脚本攻击，攻击者通过注入恶意的脚本，再用户浏览网页的时候进行攻击，比如获取cookie，或者其他用户身信息。

    防御方式：设置httpOnly属性，对用户的输入进行检查，进行特殊字符过滤

    # 37.如何查看网站的性能

    • 被动测试：再被检测的页面置入脚本或者探针，当用户访问网页时，探针自动采集数据并传回数据库进行分析
    • 主动检测：主动的搭建分布式受控环境，模拟用户发起页面的访问请求，主动采集性能数据并进行分析。

    # 38.cookie有哪些字段可以设置

    name： cookie的名称

    value： cookie的值

    domain： 访问此cookie的域名

    path： 可以访问此cookie的页面路径

    expires/Max-Age： 此cookie的超时时间

    Size： cookie的大小

    http： cookie的httponly属性，若为true则只有再http请求头中会带有此cookie的信息，document.cookie则访问不到cookie

    secure： 设置是否只能通过https来传递此条cookie
